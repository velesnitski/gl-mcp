# 004 – AI Adoption HTML Report

## Status
Accepted

## Context
`get_ai_adoption` (ADR 001–003) produces a markdown scorecard sized for chat
consumption. Townhall and leadership audiences need the same data as a
self-contained, presentable artifact: open in a browser, export to PDF, no
markdown renderer required. The repo already ships three HTML reports
(`generate_dev_report`, `generate_team_report`, `generate_project_report`)
with an established dark theme, shared print CSS, and an Export PDF button.

Two design questions:

1. **Where does the HTML live?** All existing HTML rendering sits in
   `reports.rs`, but the adoption report is a pure projection of
   `RepoMarkers`/`RepoResult` data and the level/trajectory/flag functions in
   `adoption.rs`. Moving the renderer to `reports.rs` would either duplicate
   the aggregation logic or force `adoption.rs` internals to be widely
   exported.
2. **How to avoid two scan implementations?** The markdown and HTML outputs
   must stay in lockstep on scan semantics (dormant skipping, batching,
   marker detection) or they will silently diverge.

## Decision
1. **Extract `scan_group`.** The scanning half of `get_ai_adoption` (project
   listing, dormant filtering, 10× concurrent `scan_repo` batching) moves
   into `pub(crate) async fn scan_group(client, group_path, days) ->
   Result<AdoptionScan>` where `AdoptionScan { group, days, active:
   Vec<RepoResult>, dormant_count }`. `get_ai_adoption` becomes scan +
   existing markdown formatting — behavior and output unchanged (all
   existing tests pass untouched). Empty-group and all-dormant cases are
   represented as an empty `active` vec and re-rendered by each formatter.
2. **HTML stays next to its data.** `generate_ai_adoption_report` lives in
   `adoption.rs`, not `reports.rs`. It reuses the shared chrome from
   `reports.rs` — `PRINT_CSS`, `EXPORT_BUTTON`, and `htmlescape` are promoted
   to `pub(crate)` — so all reports keep one print stylesheet and one export
   button.
3. **Report structure** (same palette as `generate_team_report`): header +
   summary cards (active / adopting / in-flight / scaling / attribution
   rate), horizontal level funnel (L3→L0 with in-flight as a disjoint row
   split out of L0), per-team table with a trajectory summary column (e.g.
   "2↑ 1→"), full adopting-repos table (level and trajectory color-coded),
   in-flight issue boxes, quality-flag issue boxes (red for stale
   config/setup unused, yellow for the rest), recommendations generated by
   the same rules as the markdown version, and a collapsible
   `<details>` methodology footnote so leadership can answer "what does L2
   mean" from the report itself.
4. **Attribution rate as a pure function.** `attribution_rate(adopting) ->
   Option<f64>` = among adopting repos with active usage, the share whose
   usage is trailer-visible (`ai_commits > 0`); `None` when nothing shows
   usage. Pure over `RepoMarkers` so it is unit-testable without a client.
5. **Wiring:** `GenerateAiAdoptionReportParams` (group_path required, days
   flex-default 30, instance), registered via `simple_tool!`. Read-only —
   not in `WRITE_TOOLS`.

## Consequences
- Easier: one scan implementation feeds both outputs, so future signal
  changes (ADR 003-style) automatically reach the HTML report; the report is
  a single self-contained file suitable for save-and-open or PDF export;
  repo/branch/team names are HTML-escaped at every interpolation point.
- Harder/limits: `RepoResult` and its helpers are now `pub(crate)` instead
  of private; the HTML body is not snapshot-tested (it embeds the current
  date) — only the pure aggregation helper is locked by tests; the report
  intentionally renders *all* adopting repos (no top-25 cap like markdown)
  since file size is not token-constrained.
- Follow-up: none planned; if a fourth output format appears, the funnel and
  team aggregation could move into pure structs the same way the scan did.
